## 概述

底层数据模型统一为 Quest（kind='session' | kind='task'），但 **UI 布局保持现有设计不变**：

- 侧边栏：Session 列表（kind='session'）+ Task 列表（kind='task'）分开展示
- 用户感知的是"会话"和"任务"，不感知"Quest"这个技术概念
- 新增：Quest 详情页内的 kind 切换按钮 + 切换弹窗

## 侧边栏设计

### Session 列表
- 排序：`updatedAt` 降序，置顶 Quest 排在最前
- 右键菜单：重命名、置顶/取消置顶、归档、删除
- 删除：直接删除，同时删除历史文件（`~/.pluse/runtime/quests/{questId}/events/`）

### Task 列表
- 排序：周期（recurring）→ 定时（scheduled）→ 一次性（once），同类型内按 `order_index`
- 每条显示：标题 + 调度描述（如"每天 09:00"）+ enabled 拨杆 + 状态
- **enabled 拨杆**：关闭时若有活跃 Run，先取消 Run 再关闭；关闭后调度器停止，Quest 仍在列表，状态显示 disabled；手动触发按钮仍可用
- **[▶ 立即执行] 按钮**：Quest 有活跃 Run 时禁用，执行完成后恢复
- 右键菜单：重命名、归档

### 归档与删除

**归档 = 软删除**，两者是同一操作，没有区别。

- 归档时：Quest 标记 `deleted=true`，从主列表移入归档区
- 归档前若有活跃 Run：弹确认对话框"正在执行中，归档将取消当前执行，确认？"，确认后先取消 Run 再归档
- 关联 Todo（`originQuestId = quest.id`）：保留，清空 `originQuestId`
- 历史文件：保留，不删除

**归档区：**
- Session 列表底部有"已归档"折叠区，默认折叠
- Task 列表底部有"已归档"折叠区，默认折叠
- 归档区内可以恢复（取消归档）或永久删除

**永久删除：**
- 只能在归档区内操作
- 物理删除数据库记录 + 历史文件目录
- 不可恢复

## Task 详情页（kind='task'）

顶部右侧有 **[转为会话]** 按钮，点击直接切换（无弹窗）。

```
任务名称                                      [转为会话] [▶ 立即执行]
──────────────────────────────────────────────────────
调度信息：每天 09:00 · AI Prompt · pending
上次执行：2026-04-17 09:01 · 成功
──────────────────────────────────────────────────────
执行历史（最近 20 条，不分页）
  ▶ 2026-04-17 09:01  成功
    └ 已完成代码审查，发现 3 处问题...（点击展开完整输出）
  ▶ 2026-04-16 09:00  成功
  ▶ 2026-04-15 09:02  失败
    └ 失败原因见执行记录（failureReason: timeout）
──────────────────────────────────────────────────────
消息历史（AI 执行产生的对话记录）
```

**执行失败的展示：**
- 失败条目显示红色状态标记
- 展开后可看到 failureReason（timeout / process_lost / error）
- 不在列表主视图显示，用户主动展开才看到

## kind 切换弹窗：session → task

点击 **[转为任务]** 后弹出：

```
┌─────────────────────────────────────┐
│ 转为任务                             │
├─────────────────────────────────────┤
│ 标题 *                               │
│ [___________________________]       │
│                                     │
│ 调度类型 *                            │
│ ( ) 一次性  ( ) 定时  (•) 周期        │
│                                     │
│ 执行时间 *                            │
│ [cron 表达式 / 时间选择器]             │
│                                     │
│ 执行内容 *                            │
│ [___________________________]       │
│                                     │
│ Agent        [Codex ▾]              │
│ 续接上下文   [✓]                      │
│ 完成后通知   [ ]                      │
├─────────────────────────────────────┤
│              [取消]  [确认转换]       │
└─────────────────────────────────────┘
```

**字段说明：**

| 字段 | 必填 | 默认值 | 验证规则 |
|------|------|--------|---------|
| 标题 | ✓ | 空 | 最多 10 字 |
| 调度类型 | ✓ | 周期 | once / scheduled / recurring |
| 执行时间 | ✓ | 空 | cron 表达式格式校验 |
| 执行内容（prompt） | ✓ | 空白，用户自己写 | 无长度限制 |
| Agent | — | Quest 当前 tool | codex / claude |
| 续接上下文 | — | 开启 | continueQuest |
| 完成后通知 | — | 关闭 | reviewOnComplete |

## kind 切换：task → session

点击 **[转为会话]** 直接切换，无弹窗。

- `kind` 改为 `session`
- 调度器暂停，但调度配置（scheduleKind / scheduleConfig / executorConfig 等）**保留在数据库**
- 切回 task 时配置仍在，无需重新填写
- AI provider context（codexThreadId / claudeSessionId）保持不变

## followUpQueue UI

参考 Codex UI 风格：

- Session 输入区上方显示排队消息列表
- 每条排队消息显示内容预览 + `[×]` 删除按钮
- 点击 `[×]` 取消该条消息，不影响当前执行中的 Run
- 队列为空时不显示该区域

## 验收标准

### 布局
- [ ] 侧边栏保持 Session 列表 + Task 列表双入口
- [ ] Session 列表显示 kind='session' 的 Quest
- [ ] Task 列表显示 kind='task' 的 Quest

### kind 切换
- [ ] Session 详情页顶部有 [转为任务] 按钮
- [ ] 点击 [转为任务] 弹出配置弹窗，标题（最多10字）/调度/执行内容为必填
- [ ] Task 详情页顶部有 [转为会话] 按钮，点击直接切换无弹窗
- [ ] task → session 切换后调度暂停，配置保留，status 重置为 pending
- [ ] session → task 切换后 AI provider context 保持不变
- [ ] Quest 有活跃 Run 时，切换按钮禁用（无 tooltip，直接禁用）

### 交互行为
- [ ] 归档有活跃 Run 时弹确认对话框，确认后先取消 Run 再归档
- [ ] enabled 拨杆关闭时有活跃 Run，先取消 Run 再关闭
- [ ] 执行失败条目展开后显示 failureReason，主视图不显示
- [ ] [▶ 立即执行] 按钮在 Quest 有活跃 Run 时禁用（无 tooltip）

### followUpQueue
- [ ] 有排队消息时，输入区上方显示队列列表
- [ ] 每条排队消息可逐个取消
- [ ] 队列为空时不显示该区域

### Todo 面板
- [ ] 侧边栏底部显示未完成 Todo 数量
- [ ] 面板内显示来源 Quest 名称
