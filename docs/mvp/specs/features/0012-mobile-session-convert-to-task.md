# 0012 Mobile Session to Task Conversion

## 背景

当前移动端在会话页点击“转换成任务”时，会直接把当前 Quest 从 `session` 改成 `task`。  
这种行为过于突兀，用户会感觉“内容还没确认就已经变了”。

问题不只是关闭弹窗后的路由回退，而是转换动作本身发生得太早。

## 目标

把“会话转任务”改成两步流程：

1. 点击转换按钮后，只打开一个设置弹窗。
2. 用户在弹窗里确认标题、描述、执行配置等内容。
3. 只有在点击保存后，才真正把当前 Quest 转成 `task`。

在未保存之前：

- 当前会话内容保持不变
- Quest 的 `kind` 保持 `session`
- 关闭弹窗后，必须回到会话页本身

## 交互要求

- 移动端会话页的转换按钮只负责“打开弹窗”
- 弹窗关闭按钮、遮罩点击、`Escape` 都只关闭弹窗，不提交转换
- 保存按钮才执行转换
- 如果转换失败，弹窗继续保留，显示错误信息

## 建议的弹窗内容

弹窗建议至少包含以下设置：

- 任务标题
- 任务描述
- AI 执行说明或提示词
- 是否继续沿用当前上下文

如果后续需要更完整的任务配置，再扩展为：

- 工具
- 模型
- 推理强度
- 计划方式

## 实现边界

### 不应该做的事

- 不要在点击转换按钮时直接调用 `updateQuest({ kind: 'task' })`
- 不要在关闭弹窗时顺带做任何状态切换
- 不要让用户为了取消转换而先“已经变成任务”

### 应该做的事

- 先打开弹窗，再把当前会话内容带入表单
- 保存时再提交 `updateQuest`
- 转换完成后，由路由和详情页自行切换到任务视图

## 代码层建议

建议把“打开转换弹窗”和“提交转换”拆开：

- `ChatView` 只负责控制弹窗开关
- `TaskComposerModal` 或单独的转换弹窗组件负责表单和提交
- 提交成功后再更新当前 Quest 数据

## 验收标准

- 移动端会话页点击转换按钮后，只弹出设置弹窗
- 关闭弹窗不会改变 Quest kind
- 只有点击保存后，当前会话才会变成任务
- 保存前取消，不会出现“内容已经转成任务但用户还没确认”的突兀感
