# 0004 — Automation IA And Naming Phase 1

**状态**: draft
**类型**: spec
**关联 requirement**: `docs/v2/requirements/0004-ai-task-result-handling.md`
**关联 design**: `docs/v2/designs/0004-automation-ia-and-naming.md`

## 本期目标

Phase 1 只做一个足够小、但用户能立刻感知到的收敛：

- 把用户面里的 `AI 任务 / 人类任务` 收敛为 `自动化 / 待办`
- 把右侧工作面板的对象筛选口径从 `人类 / AI` 收敛为 `待办 / 自动化`
- 把 `scheduleKind='once'` 的用户文案正式收敛为 `手动`

本期不试图解决“已处理”如何落模型，也不试图重构底层 Quest / Run / Todo 命名。

## 本期范围

### 1. 用户面命名收敛

所有用户可见的主文案按以下映射收敛：

| 旧文案 | 新文案 |
|---|---|
| AI 任务 | 自动化 |
| 人类任务 | 待办 |
| 创建 AI 任务 | 创建自动化 |
| 创建人类任务 | 创建待办 |
| 新 AI 任务 | 新自动化 |
| 转为任务 | 转为自动化 |

这次只收敛用户面，不要求内部变量名、组件文件名同步改掉。

### 2. 工作面板筛选收敛

右侧统一工作面板的顶层筛选统一改为：

- `待办`
- `自动化`

不再出现：

- `人类 / AI`

对应的空态文案、按钮文案、aria-label、title 也同步改口。

### 3. 详情页与弹窗口径收敛

以下入口与详情页口径统一调整：

- 自动化详情页 kicker：`自动化`
- Todo 详情页 kicker：`待办`
- 会话中的 kind 切换动作：`转为自动化`
- 从自动化详情创建 Todo 的动作：`新建待办`

如果当前仍使用统一创建弹窗，本期允许继续保留一个弹窗承接两类对象，但：

- 弹窗内对象类型必须明确显示为 `待办` 或 `自动化`
- 不再使用 `AI 任务 / 人类任务`
- 当用户已在 `待办` 或 `自动化` 筛选下时，创建按钮优先使用对应文案

### 4. 调度类型的显示收敛

所有用户可见的调度类型文案统一按以下规则显示：

- `scheduleKind='once'` → `手动`
- `scheduleKind='scheduled'` → `定时`
- `scheduleKind='recurring'` → `周期`

本期只做显示层映射，不改动数据库值和类型定义。

## 视图与交互规则

### 1. 右侧工作面板

当前面板继续作为“待办 + 自动化”的统一入口，不拆新页面。

Phase 1 只修改：

- 顶层筛选标签
- 当前筛选下的空态文案
- 新建入口文案
- 自动化卡片的对象标识文案

不在本期引入新的列表分区、排序逻辑或结果处理状态分组。

### 2. 自动化详情页

`Quest(kind='task')` 详情页只做口径收敛：

- 顶部对象标识改为 `自动化`
- 相关按钮和说明文案同步改口

不在本期调整字段布局、信息分组或结果处理展示。

### 3. Todo 详情页

Todo 详情页只做对象标识收敛：

- 顶部对象标识改为 `待办`

不在本期改动 Todo 的字段、状态流转或交互结构。

## 模块影响范围

预计至少涉及以下模块：

- `packages/web/src/views/components/TodoPanel.tsx`
- `packages/web/src/views/components/TaskComposerModal.tsx`
- `packages/web/src/views/components/TaskDetail.tsx`
- `packages/web/src/views/pages/MainPage.tsx`
- `packages/web/src/i18n.tsx`

如存在其他直接暴露 `AI 任务 / 人类任务 / 转为任务` 的用户面文案，也应一并收敛。

## 数据与接口边界

### 1. 本期不做 schema 变更

本期不新增、不删除任何数据库字段。

特别是以下概念继续保留：

- `Quest.kind='task'`
- `scheduleKind`
- `Run.trigger`
- `executorKind`

### 2. 本期不做 HTTP / CLI 改名

即使用户面改叫“自动化”，本期仍允许：

- API 路由继续使用 `quests`
- CLI 命令继续使用 `quest`
- 代码内部继续使用 `task` 作为部分技术命名

## 不在本期范围

以下内容明确不做：

- “已处理 / 待处理结果” 的状态设计
- 自动化卡片的信息层级重构
- executor 维度的新筛选
- Quest / Run / Todo 的内部命名重构
- 旧类型文件和历史文档的全面清理
- 文件级组件重命名
- 新的页面或新的导航结构

## 验收标准

本期完成后，应满足：

1. 用户面不再出现 `AI 任务 / 人类任务` 作为对象主命名  
2. 右侧工作面板对象筛选统一显示为 `待办 / 自动化`  
3. 自动化详情页和待办详情页的对象标识分别正确显示为 `自动化 / 待办`  
4. `scheduleKind='once'` 在所有用户可见位置统一显示为 `手动`  
5. 不引入任何 schema、API、CLI 的破坏性改动  

## 实施顺序建议

1. 先统一 `i18n` 文案和映射表  
2. 再修改工作面板、详情页、创建弹窗中的对象命名  
3. 最后做一次全文案回归，清掉遗留的 `AI 任务 / 人类任务 / 转为任务` 用户面文案
