# 0001 — Human Workload Control Phase 1

**状态**: draft
**类型**: spec
**关联 requirement**: `docs/v2/requirements/0001-human-workload-control.md`
**关联 design**: `docs/v2/designs/0001-human-workload-control.md`

## 本期目标

Phase 1 只做基础承接层，目标是先解决以下三个最直接的问题：

- 人类需要重复解释上下文
- 人类需要手动收拾 Agent 丢出来的复杂度
- Agent 会把原始复杂度直接甩给人，导致人被迫接管流程

本期不追求完整的负载控制、托管和动态训练，只做最小但高杠杆的承接能力，并尽量复用现有 `Quest / Run / Todo` 边界。

## 本期范围

### 1. 基于现有 Todo 的结构化人类请求

当 Agent 需要人类参与时，系统不应直接暴露原始日志、零散 review 或多个碎 Todo，而应生成单个结构化交接单元。

Phase 1 只支持两类阻塞型人类请求：

- 缺关键信息
- 缺审批

这两类请求的判断口径应固定如下：

- `缺关键信息`
  - Agent 缺少一个只能由人类提供的输入
  - 没有这个输入，当前 Quest 无法继续推进

- `缺审批`
  - Agent 已经知道下一步怎么做
  - 但这一步必须由人类授权或承担责任，因此不能自行继续

这个交接单元至少应表达：

- 现在为什么要找人
- 需要人做什么
- 这次参与属于哪一类
- 对应哪个 Quest / Run
- 最小必要上下文是什么

Phase 1 不新增新的顶层持久化对象，而是复用现有 `Todo` 作为承载对象，但必须把它从“普通待办”提升为“结构化人类请求”。

### 2. 人类请求的最小闭环

Phase 1 必须明确人类请求的最小流转，而不是只定义它“长什么样”。

最小闭环至少包括：

- 创建请求
- 同一问题优先更新现有请求，而不是重复创建
- 人类处理后请求可被标记为已解决
- 请求解决后，Quest / Run 回到可继续推进的状态

这一步的目标是先止住“Todo 洪水”、避免人类重复收拾残局，并让人类参与真正形成闭环。

### 3. 按需生成当前上下文

Phase 1 不维护一个单独的持久摘要对象，而是在进入当前工作面时，基于现有对象按需生成当前上下文。

当前上下文至少应由以下来源组成：

- Quest 当前标题 / 描述 / 状态
- 最新 Run 的相关结果
- 当前未解决的人类请求
- 与当前请求直接相关的最小必要说明

这样做的目标是：

- 降低重复解释成本
- 让人类快速重新进入当前状态
- 避免引入一个会很快过期的摘要维护系统

### 4. 复用现有 Human 入口

Phase 1 不完整重构 Human side，只要求至少在一个现有入口中优先展示：

- 结构化人类请求
- 当前上下文

而不是把原始运行细节直接当作人类工作项。

## 不在本期范围

以下内容明确留到后续 phase：

- 全局负载控制与复杂优先级重排
- 长期任务保护策略
- 缺方向判断类请求
- 高风险暂停类请求
- 周期任务识别、一键动作和大规模托管
- 动态训练 / 教练模式
- 更高阶的信任、解释和纠偏机制
- 跨 Project 的工作面统筹
- 独立的长期摘要维护机制

## 关键实现约束

### 1. 用户侧概念极简优先

Phase 1 的首要原则不是机械复用现有对象，而是尽量减少用户需要理解的新概念数量。

因此本期默认优先复用现有：

- `Quest`
- `Run`
- `Todo`

但如果新增一个概念能同时压缩多个需求、显著减少用户接收的信息量，并让 Agent / Human 协作更顺，则允许引入新的内部承载方式。

判断标准应是：

- 是否同时解决多个需求
- 是否减少用户侧认知负担
- 是否改善 Agent 与 Human 的协作效率

### 2. 不引入独立摘要对象

Phase 1 先通过现有 `Quest / Run / Todo` 组合出当前上下文，而不新增独立摘要持久化对象。

如果后续证明按需生成无法满足需求，再在后续 phase 讨论是否引入更稳定的摘要机制。

### 3. 先做交接质量，不先做聪明排序

Phase 1 的成功不取决于排序算法是否更聪明，而取决于：

- 交接是否被结构化
- 请求是否形成闭环
- 当前上下文是否能按需进入

### 4. 先降低人类进入成本

本期要优先解决“进入当前状态需要重新理解太多东西”的问题，而不是追求更复杂的自动化广度。

## 交付物

本期 spec 对应的实现应交付：

1. 一套基于现有 `Todo` 的结构化人类请求机制
2. 一套人类请求最小闭环规则
3. 一套基于现有 `Quest / Run / Todo` 的按需上下文生成方案
4. 至少一个人类可用入口，能看到：
   - 当前结构化请求
   - 对应 Quest / Run
   - 最小必要上下文

## 验收标准

Phase 1 完成后，应满足以下标准：

- 同一 Quest 的同类阻塞不会持续生成多个碎请求
- 每个人类请求都明确说明：
  - 为什么找人
  - 需要人做什么
  - 对应哪个 Quest / Run
- 人类处理完请求后，该请求可以被明确关闭，且 Quest / Run 回到可继续推进的状态
- 用户无需回看完整历史，也能快速理解当前 Quest 在做什么
- 原始运行输出不再直接等于人类工作项
- Phase 1 的改动尚未引入复杂优先级系统、托管系统、动态教练系统或独立摘要对象

## 实施顺序建议

建议按以下顺序落地：

1. 先建立基于现有 `Todo` 的结构化人类请求
2. 再补人类请求的最小闭环
3. 再基于现有 `Quest / Run / Todo` 组合按需上下文
4. 最后把一个现有 Human 入口改成优先展示结构化请求和当前上下文

这个顺序的原因是：

- 没有结构化请求，就没有稳定的人类交接载体
- 没有闭环，请求仍然只是换了外观的 Todo
- 没有按需上下文，人类仍需要重新理解完整历史
