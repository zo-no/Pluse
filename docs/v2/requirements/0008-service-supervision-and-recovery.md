# 0008 — 服务常驻与自动恢复

**状态**: draft
**类型**: requirement
**优先级**: high

## 背景

Pluse 当前主要通过 `pnpm dev` 或 `pnpm start` 启动服务。这个启动方式适合开发调试，但它依赖当前终端进程。

当终端关闭、父进程被杀、机器重启、开发进程异常退出时，Web 页面和 API 会同时不可用。用户在浏览器里看到的是连接失败，而不是一个可理解、可恢复的系统状态。

## 真实问题

Pluse 已经承担远程 AI 工作台职责，但服务进程仍像普通开发进程一样脆弱：

- 服务是否还在运行需要用户手工判断
- 服务退出后不会自动恢复
- 用户不知道是端口冲突、进程退出、构建缺失，还是后端真正崩溃
- Agent 或用户需要恢复工作时，没有一个稳定的服务管理入口

## 顶层目标

Pluse 需要具备本机常驻运行能力：服务应能在用户登录后自动启动，在异常退出后自动恢复，并提供清晰的状态查询和手动控制入口。

## 核心需求

### 1. 服务脱离临时终端

Pluse 的长期运行方式不应依赖当前 Codex 会话、shell 窗口或一次性的 `pnpm dev` 前台进程。

### 2. 异常退出可自动恢复

当服务进程异常退出时，系统应自动重新拉起，避免用户刷新页面后直接不可用。

### 3. 状态可诊断

用户和 Agent 应能快速知道：

- 服务是否安装为常驻服务
- 服务当前是否在运行
- 当前监听端口是什么
- 最近一次启动时间和进程 ID
- 健康检查是否通过

### 4. 手动控制入口明确

用户应有稳定命令控制服务：

- 启动
- 停止
- 重启
- 查看状态
- 查看日志位置

### 5. 开发模式和常驻模式边界清晰

`pnpm dev` 仍用于开发和热更新；常驻模式用于日常使用，不应让用户混淆两者。

## 成功状态

若该需求被满足，用户应能感受到：

- 重启机器或关闭终端后，Pluse 仍能恢复可用
- 服务异常退出后会自动恢复
- 页面不可用时，可以用一个标准命令判断原因
- Agent 不需要反复手工查端口、杀进程、重启前台服务

## 不在范围内

当前 requirement 不要求：

- 多用户部署和远程服务器运维
- Docker / systemd / Cloudflare 等跨平台部署方案
- 进程级高可用集群
- 数据库自动修复
- 前端离线编辑和离线缓存
- 自动修复所有业务运行失败

## 讨论结果

本需求优先解决 macOS 本机长期使用场景。开发模式保留，但不再把开发模式当作日常常驻方案。
