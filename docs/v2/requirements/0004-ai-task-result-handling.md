# 0004 — Todo / Automation Boundary

**状态**: draft
**类型**: requirement
**优先级**: high

## 背景

Pluse 已经在底层把两类对象分开了：

- `Todo`：人工待办
- `Quest(kind='task')`：自动化

但产品层的语义和信息架构还没有完全跟上这条边界。

当前问题主要有三类：

1. 自动化仍然会被产品文案误导成“可完成的任务”
2. 自动化会被塞进 `已完成 / 已处理` 这类人类工作流语义
3. 右侧工作面板仍然带有 `会话` 级别的 scope 视角，不符合用户实际并行处理工作的方式

这会让用户对系统对象产生错误预期：

- 以为自动化跑完一次就“结束了”
- 以为自动化也需要像待办一样被人工勾掉
- 以为当前只能围绕一个会话来处理工作

## 核心问题

Pluse 缺少一套足够清晰的产品语义来回答：

- 什么是 `待办`
- 什么是 `自动化`
- 哪些状态只属于待办，哪些状态不应出现在自动化上
- 工作面板应该围绕“当前工作项”组织，还是继续围绕“当前会话”组织

## 需求目标

系统需要在用户面稳定表达以下边界：

1. `Todo` 是给人处理的工作项
2. `Automation` 是系统持续存在的执行能力
3. `完成 / 已处理` 属于待办语义，不属于自动化本体语义
4. 工作面板应优先服务“并行处理多个工作项”，而不是“只围绕一个会话处理”

## 关键要求

### 1. 自动化没有“已完成”概念

自动化可以有：

- 是否启用
- 如何触发
- 最近一次运行状态

但不能因为成功运行过一次，就在产品上被表述为“已完成”。

### 2. 自动化没有“已处理”概念

“已处理”是人类工作流语言，适用于：

- 待办是否已经被人处理完

不适用于：

- 自动化本体是否仍然存在
- 自动化是否还可继续触发

如果自动化产出需要人类跟进，应上浮为 Todo，而不是给自动化本体增加“已处理”状态。

### 3. 工作面板不再提供会话级 scope 作为主视角

用户处理工作时，不会一次只处理一个会话。

因此右侧工作面板不应继续强调：

- `全局 / 项目 / 会话`

至少不应把 `会话` 作为主视角之一。

工作面板应该优先围绕：

- 待办
- 自动化
- 项目 / 全局
- 合理的对象分组

来组织。

### 4. 自动化需要自己的分组方式

自动化不能继续只是混在统一列表里，再通过“是否已完成”进入历史区。

自动化至少需要按自身语义分组，例如：

- 手动
- 定时
- 周期
- 运行中
- 异常

具体分组方案属于 design / spec，但 requirement 要先锁定一条原则：

自动化必须按自动化语义分组，而不是按待办语义分组。

### 5. 待办和自动化可以出现在同一个工作面，但不共享同一套生命周期语义

两者可以共存于右侧工作面板，但：

- 待办保留 `待处理 / 已完成 / 已取消`
- 自动化保留 `运行中 / 待触发 / 失败 / 已取消 / 已执行`

它们不能再因为共用一个列表，就被强行解释成同一类对象。

## 不在本需求范围内

当前 requirement 不规定：

- 是否调整数据库 schema
- 是否删除 `Quest.status='done'`
- 是否引入新的 automation 字段
- 是否重命名 `reviewOnComplete`
- notification / inbox / result-attention 的完整体系

这些都属于后续 design / spec 或更大需求。

## 成功状态

如果这个需求被满足，用户应能稳定感受到：

- `待办` 是给人完成的
- `自动化` 是持续存在的系统能力
- 自动化不会再因为跑完一次就被说成“已完成”
- 自动化不会再落入“已处理”这种人类语义
- 工作面板在实际使用上更像工作项面板，而不是会话附属面板

## 失败状态

如果后续设计或实现仍然出现以下情况，说明这个需求没有被满足：

- 自动化成功运行一次后，被主文案或主分组表述为“已完成”
- 自动化被放进 `已处理` 历史区
- 用户仍被引导用“勾掉任务”的方式理解自动化
- 工作面板仍然默认要求用户从 `会话` 视角处理工作
