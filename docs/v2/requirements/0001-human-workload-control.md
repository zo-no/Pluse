# 0001 — Human Workload Control

**状态**: draft
**类型**: requirement
**优先级**: high

## 背景

Pluse 目前已经可以记录 Todo、管理 Quest、承载多个 Project，并允许 Agent 在运行过程中创建需要人类介入的待办。

但随着并行事项变多，系统暴露出一个更底层的问题：

- 用户同时背负的事情太多
- Todo 能收集任务，但不能控制人类承载量
- Agent 持续把事情推给用户，容易让系统变成“更有条理的混乱”

因此当前需要优先解决的，不是某个单点 Todo 交互，而是“人类工作负载控制”问题。

## 顶层目标

这个需求的顶层目标不是“把 Todo 做得更强”，而是：

- 让用户在持续使用中越来越轻
- 让长期重要事务被越来越稳定地保护
- 让系统越来越能接管重复、低价值、可模板化的工作

换句话说，Pluse 应该逐步把用户从低价值人类劳动中解放出来，让人把注意力更多放在真正重要的长期任务上。

## 顶层原则

这个需求后续无论如何设计或分期实现，都应遵循以下原则：

- 目标推进优先于流程完整
- AI 是默认主推进者，人类是高价值例外参与者
- 系统必须按 Agent 与人类的不同速度、不同职责、不同带宽设计
- 人类参与应被压缩并包装成最小、最清晰、最高价值的交互单元
- 系统应优先吞掉重复、低意义、可预测的劳动
- 系统应在必要时动态训练人类更适应与 AI 协作，但训练必须服务于整体推进
- 系统必须可托付、可纠偏，而不是只能强行自动化
- 整体演化必须是渐进的，而不是一口气把人从流程里抽掉
- 控制减少不等于主导感丢失，用户仍应保有方向感和最终主导权

## 问题定义

当前用户面临的核心问题不是“不知道有什么事”，而是：

1. **并行过载**
   - 同时进入用户视野的事项过多
   - 用户无法稳定只处理少量当前任务

2. **优先级失真**
   - 太多事情都看起来重要
   - 缺少稳定的降级、延后、舍弃机制

3. **Agent 派活无统一约束**
   - 各个 Agent 都可能创建 Todo
   - 系统缺少统一准入、去重、合并和通知门槛

4. **当前承诺面不清晰**
   - 用户需要的是“现在到底做什么”
   - 不是一个更长、排序更复杂的待办列表

5. **系统不会让用户越来越轻**
   - 当前 Todo 只会累加工作，不会主动识别哪些工作应该转成周期任务、快捷动作或 AI 托管任务
   - 用户缺少“做着做着，系统越来越接管”的感受

## 核心需求

系统需要具备以下能力：

### 1. 系统控制的是负载，而不是任务数量

系统不应只按“任务条数”管理用户，而应按真实负载管理用户。

这里的负载包括：

- 打断成本：是否会把用户从当前工作中拉走
- 上下文切换成本：是否需要重新进入另一个工作语境
- 占坑成本：即使暂时不做，也会持续占用用户心智
- 深度时间成本：是否需要连续、完整的一段时间才能推进
- 压力成本：不处理是否会快速制造截止风险、关系风险或外部承诺风险
- 长期挤压成本：是否会持续挤占长期重要任务的推进空间

系统需要限制进入用户当前工作面的总负载，避免工作面无限膨胀。

### 2. 帮助用户形成明确的当前承诺

系统应能让用户清楚知道：

- 现在做什么
- 接下来做什么
- 哪些事情暂时不做

### 3. 对 Agent 的人类介入请求做统一管理

当 Agent 认为需要人类介入时，系统不应直接把请求变成用户眼前的新任务，而应先统一评估。

### 4. 重要性应采用“人定长期方向 + AI 做局部判断”

“重要”不应完全由用户手工逐条标注，也不应完全由 AI 自行猜测。

系统应采用分层判断方式：

- 用户定义长期方向和稳定的重要性信号
- AI 基于当前上下文做局部重要性判断

这样既能保护长期目标，又能让系统在局部情境下动态判断什么值得优先处理。

默认情况下，AI 的局部重要性判断应至少参考以下信号：

- 是否阻塞当前主线项目、P0 Quest 或关键执行路径
- 是否直接推动长期重要目标，而不是只解决局部噪音
- 是否涉及外部承诺、明确时间窗口或别人正在等待
- 是否能显著解锁后续工作，避免更多任务持续堆积
- 是否只是“容易立刻做”的小事，但对主线推进价值很低

### 5. 优先级需要可被重排，而不是只被记录

系统应支持随着上下文变化，对当前工作面进行重排、降级和清退。

### 6. 用户需要“轻”，而不是“全”

系统的成功标准不是记录尽可能多的任务，而是让用户在任何时刻只承载自己真正能推进的少量事项。

### 7. 系统应逐步识别并托管可重复工作

系统不仅要管理当前任务，还应逐步识别：

- 哪些工作是周期性的
- 哪些工作可以抽成一键触发动作
- 哪些工作可以默认由 AI 托管

从而让用户承担的即时、重复、低价值任务持续下降。

默认迁移路径应为：

- AI 识别重复模式或固定流程后提出托管建议
- 用户对这类工作做一次确认
- 后续同类工作默认进入周期化、快捷动作或 AI 托管执行

对于明显低风险、模式固定、输入输出清晰的工作，系统也可以更积极地提出迁移建议。

### 8. 托管模式默认是边界内自主执行

系统的目标不是成为只会给建议的助手，也不是每次执行前都要求确认。

默认托管模式应为：

- 在明确边界内，AI 默认自主执行
- 只有在超出边界时，才请求人类介入

这类边界通常包括：

- 是否低风险
- 是否重复出现
- 是否具有明确输入输出
- 是否不涉及主观判断
- 是否不需要审批

### 9. AI 托管后，默认只在例外情况下打断用户

当任务被 AI 接管后，系统不应继续把完整执行过程暴露给用户。  
只有在以下场景下，才应再次请求人类介入：

- 需要人提供缺失信息
- 需要人做审批
- 需要人做关键判断
- AI 执行出现异常或卡住

## 当前收敛决议（2026-04-22）

以下决议用于锁定后续 design / spec 的方向，避免需求漂移：

1. **即时处理面的容量上限**
   - 系统默认同屏只暴露少量“立即处理项”
   - 当前默认上限为 **3 项**
   - 超出部分应进入后续队列，而不是继续并行暴露给人

2. **Waiting 请求与普通 Todo 分离**
   - 需要人类信息、审批或关键判断的请求，不与普通待办混在同一语义层
   - Waiting 请求应作为独立类别承接，避免“通知、计划、执行事项”全部挤在 Todo 一处

3. **低重要事项默认批处理**
   - 对低重要且可延后事项，系统默认进入批处理窗口
   - 目标是降低高频打断，而不是让每个小事项都立即占用用户注意力
   - 批处理窗口的具体规则属于后续 design / spec 阶段

## 不在范围内

当前需求文档不讨论：

- 具体数据库字段怎么设计
- CLI / API 叫什么名字
- UI 长什么样
- 是否拆成独立进程或独立产品

这些内容属于后续 design / spec 阶段。

## 成功状态

如果这个需求被满足，用户打开系统时应该能感受到：

- 眼前不是一大堆 pending 事项
- 系统已经替自己做了一轮过滤
- 当前真正需要承担的事情很少、很清楚
- Agent 不会无限制地把压力直接转移给自己
- 自己知道为什么现在做这些事，而不是别的事
- 长期重要任务能更稳定地持续推进，而不是总被噪音挤掉
- Agent 等待人类的时间明显减少
- 自己被低价值事项打断的频率明显下降
- 很多重复工作正在逐步从“人来做”转向“系统托管”
- 系统只有在真正需要人类时才会上浮提醒
- 人类参与虽然更少，但每次参与都更有价值
- 自己越来越会和 AI 协作，而不是每次都重新磨合

并且具备以下可观察信号：

- 在默认模式下，用户当前工作面中的“立即处理项”长期稳定在小容量（默认上限 3）
- Waiting 请求不再和普通 Todo 混成同一语义队列
- 低重要、可延后事项显著减少即时打断，更多在批处理窗口内集中处理

## 失败状态

如果后续功能实现仍然出现以下情况，说明这个需求没有真正被满足：

- Todo 只是字段更多、排序更复杂
- 用户仍然需要手动收拾 Agent 产生的大量待办
- 系统仍然展示一个庞大的 pending 列表
- 系统只能提醒更多事，不能减少用户承载
- 周期性或重复性工作长期停留在人手里，没有逐步托管出去
