# 0010 — Project Priority Tiers And Visibility

**状态**: draft
**类型**: requirement
**优先级**: high

## 背景

Pluse 已经在提醒系统里引入了项目级优先级，用来区分：

- `mainline`
- `priority`
- `normal`

这让提醒池在跨项目排序时，能够更好地保护主线项目。

但当前这套语义仍停留在提醒侧，存在几个明显缺口：

- 项目本体没有同等优先级字段，用户需要进入提醒视图才能看到差异
- “主线 / 优先 / 普通”之外，没有稳定的“低优先级停放区”
- 暂不推进但不应归档的项目，只能和普通项目混在一起
- 项目入口、项目列表、提醒分组之间没有统一口径

当用户开始按“人生优先级排序”管理项目时，这个缺口会直接削弱系统的主线保护能力。

## 问题定义

当前核心问题不是“项目列表少一个标签”，而是：

1. **项目级长期重要性没有统一承载**
   - 提醒模块已经有项目优先级
   - Project 自身却没有同一套稳定语义
   - 结果是系统排序依据和用户看到的项目入口脱节

2. **缺少“低优先级但不归档”的停放层**
   - 有些项目并非结束，也不该归档
   - 但它们当前不该持续占据用户默认注意力
   - 现有 `normal` 过于拥挤，无法表达“先放着，但保留在线”

3. **默认工作面没有明确保护主线**
   - 如果项目入口看起来都差不多，用户会重新靠记忆判断轻重
   - 这会削弱 Pluse 在负载控制上的系统作用

4. **提醒口径与项目口径分裂**
   - 提醒排序已按项目优先级工作
   - 项目列表、项目切换器、项目分组却没有同样的结构与文案
   - 同一个项目在不同入口可能呈现出不同重要性信号

## 核心需求

系统需要建立一套统一的项目优先级能力，满足以下要求：

### 1. 项目优先级应成为 Project 的稳定语义，而不是提醒模块私有配置

项目优先级应直接描述“这个项目在用户长期推进结构中的位置”，而不是只为某个列表排序服务。

这套语义至少需要覆盖四档：

- `mainline`
- `priority`
- `normal`
- `low`

### 2. `low` 必须表达“暂不推进，但不归档”

`low` 不是：

- 已完成
- 已废弃
- 已归档

`low` 表达的是：

- 项目仍保留在线
- 历史、自动化、附件、上下文都继续存在
- 但默认不应与主线、优先、普通项目争夺同等注意力

### 3. 默认入口需要主动降低低优先项目的存在感

低优先级项目应继续可见、可访问、可恢复推进，但默认不应与更高优先级项目平铺混排。

系统需要至少支持：

- 在提醒视图中按低优先级单独分层
- 在项目分组或项目入口中默认折叠低优先级项目

### 4. 项目入口必须直接透出优先级

用户无需进入提醒面板，也应能在项目列表、项目切换器或其他主要项目入口中直接看出：

- 哪个是主线
- 哪些是优先
- 哪些是普通
- 哪些是低优先

### 5. 排序、筛选、文案和折叠行为必须统一

项目优先级不是一个孤立字段，而是一套跨入口行为语义。

至少需要统一：

- 默认排序
- 分组顺序
- 默认折叠规则
- 筛选口径
- 用户文案 / badge / label

### 6. 现有 reminder project priority 需要明确迁移关系

当前提醒模块已经保存项目优先级。

新能力必须明确：

- 现有数据如何迁移或收敛
- API 如何兼容过渡
- 提醒排序是否改为读取 Project 统一优先级
- 旧的 reminder project priority 配置是否继续保留

## 与上位需求的关系

这个需求属于 `0001-human-workload-control` 的项目级落地能力。

它解决的是负载控制中的一个基础问题：

- 系统如何在跨项目层面保护长期主线
- 系统如何给“暂不推进但不归档”的项目一个低打断停放层

它不是单纯的视觉增强需求，也不是提醒模块的局部重构。

## 不在范围内

当前 requirement 不讨论：

- 具体数据库字段放在 `projects` 还是独立配置表
- HTTP / CLI 接口命名
- 具体 badge 颜色和视觉样式
- 更复杂的项目评分算法
- 自动根据 Run / Todo / Reminder 数量推断优先级
- 归档语义的重定义

这些内容进入后续 design / spec。

## 成功状态

如果这个需求被满足，用户应能直接感受到：

- 项目入口不再是一长串同权重列表
- 主线、优先、普通、低优先的项目层级一眼可见
- 暂不推进的项目有稳定停放区，不必归档也不必持续打扰
- 提醒视图和项目入口对同一项目给出一致的重要性信号
- 系统能更稳定地保护长期主线，而不是把所有项目都摊平

并具备以下可观察信号：

- 项目级优先级统一为四档：`mainline / priority / normal / low`
- 低优先级项目在默认项目入口中不与高优先级项目同等展开
- 项目列表与提醒分组对同一项目展示一致的优先级标签
- 至少一个主要项目入口支持直接按优先级识别项目

## 失败状态

如果后续实现仍然出现以下情况，说明这个需求没有被真正满足：

- `low` 只是提醒面板里多了一个下拉选项
- 项目列表仍看不出项目轻重
- 低优先级项目仍默认与普通项目一起展开
- 提醒排序和项目入口继续使用两套不同的优先级口径
- 用户仍需要靠记忆判断哪些项目是主线、哪些只是停放
