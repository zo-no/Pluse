# 0003 — Agent 驱动的会话分类能力

**状态**: draft
**类型**: requirement
**优先级**: high

## 背景

Pluse 当前已经能承载大量 `Quest(kind='session')`，并通过置顶、搜索、归档和时间排序帮助用户回到最近上下文。

但当一个 Project 内会话越来越多时，系统会暴露出一个新的问题：

- 会话列表只有时间语义，没有稳定的主题语义
- 用户若想自己维护会话分组，会被迫承担额外整理劳动
- Agent 明明最了解当前会话的主题、阶段和用途，却没有稳定的系统承接方式把这种判断写回产品

这说明当前缺的不是“一个更复杂的手动分组 UI”，而是一个可被 Agent 使用的**会话分类能力**。

## 顶层目标

Pluse 需要提供一个稳定的、可持久化的会话分类承接层，使 Agent 能在推进工作时对会话进行分类，而人类不需要承担持续整理会话的负担。

这个能力的目标不是把会话变成更复杂的标签系统，而是让：

- 会话空间更容易导航
- Agent 更容易把语义结构沉淀进系统
- 人类更容易理解当前 Project 里各类会话分别在承载什么工作

## 顶层原则

- 分类能力优先，不预置方法论
- Agent 是默认分类执行者，人类不是默认整理者
- 产品负责承接分类状态，不负责内置固定分类体系
- 分类必须是低成本、可撤销、可重写的
- 分类不能破坏 Quest 统一模型、kind 切换和跨项目移动语义

## 问题定义

当前的核心问题不是“会话太多”，而是“会话缺少可持续的语义结构”。

### 1. 时间排序不能替代语义组织

当前会话主要按时间和置顶展示，这能解决“最近做了什么”，但不能解决：

- 哪些会话属于同一主题
- 哪些会话是临时探索，哪些会话是长期工作台
- 当前 Project 内有哪些稳定工作流

当自动分类开启后，正常完成首轮会话的 session 不应长期停留在 `未分组`；否则未分组区会持续膨胀，分类能力会失去产品价值。

### 2. 手动分组会把整理劳动重新压回人类

如果系统只提供“人来新建分组、移动会话”的能力，那么用户需要持续承担：

- 命名分类
- 维护分类
- 决定会话放哪
- 处理新增和漂移

这与 Pluse 让 Agent 多承担、让人类少收拾的方向相违背。

### 3. Agent 缺少把结构化判断写回系统的承接层

Agent 在长期运行中天然会形成判断：

- 这个会话属于哪个主题
- 它是在做探索、执行、复盘还是杂项支持
- 它和哪些同类会话属于同一簇

但当前系统没有稳定对象承接这种判断，导致这些语义只能停留在 Agent 的瞬时上下文里。

### 4. 当前会话空间难以形成稳定导航面

没有分类后，会话列表更像日志流，而不是工作结构。

这会同时影响：

- 人类进入当前 Project 的速度
- Agent 对长期上下文的表达能力
- 会话空间在长期使用中的可管理性

## 核心需求

系统需要具备以下能力：

### 1. 提供可持久化的会话分类承接层

系统必须允许会话拥有稳定的分类归属，并在产品中持久保存。

### 2. 允许 Agent 创建和维护分类

系统必须向 Agent 开放能力，使其可以：

- 创建分类
- 更新分类
- 删除分类
- 将会话归入某个分类
- 将会话移出分类

### 3. 分类体系应由 Agent / Human 决定，而不是产品预置

产品不应强制内置固定分类法，例如：

- “工作 / 学习 / 杂项”
- “探索 / 执行 / 复盘”

这些命名和使用方法，默认应属于 Agent 策略或人类偏好，而不是产品内核强约束。

### 4. 人类不应被要求持续手动整理

这个能力的成功标准不是“人也能手动维护”，而是“即使人不维护，系统也能通过 Agent 保持基本有序”。

### 5. 分类必须与现有 Quest 语义兼容

分类能力不能破坏：

- `Quest` 统一模型
- `session -> task -> session` 的 kind 切换
- Quest 跨项目移动
- 归档与恢复
- provider context 跟随 Quest 保留

### 6. 会话导航面需要消费这个分类能力

分类如果只存在数据库里而不进入导航面，就不能真正改善会话空间。

系统至少需要让会话列表可以基于分类组织展示。

## 产品边界

这个需求的边界应明确如下：

- **Pluse 负责**
  - 提供分类对象与归属关系
  - 提供持久化、校验和展示
  - 提供 Agent 可调用的系统能力

- **Agent 负责**
  - 何时创建分类
  - 使用什么分类体系
  - 分类门槛是什么
  - 何时重分类
  - 如何避免分类漂移
  - 不确定时如何处理

换句话说，Pluse 负责承接层，Agent 负责分类方法。

## 不在范围内

当前 requirement 不讨论：

- 分类对象的具体字段
- API / CLI 命令名
- UI 细节与交互稿
- 是否需要颜色、图标、拖拽
- 自动后台分类器是否常驻运行
- 多标签、多父子层级分类
- 跨 Project 的统一会话分类体系
- Project 上层分组能力

这些问题留到后续 design / spec。

## 成功状态

如果这个需求被满足，用户打开一个 Project 时应该能感受到：

- 会话列表不再只是时间流
- 会话已经被沉淀成少量稳定类别
- 自己不需要亲自做大量整理动作
- Agent 能把它对会话空间的理解写回系统
- 长期会话和临时会话不再混成一团

## 失败状态

如果后续实现仍出现以下情况，说明这个需求没有真正被满足：

- 产品只增加了一个手动分组 UI
- 人类仍然要持续整理会话
- Agent 无法稳定写入和更新分类
- 会话分类只存在内部字段，不进入用户导航面
- 分类体系被产品写死，Agent 只能套用固定模板
