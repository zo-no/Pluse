# 0005 — Quest 生命周期关联对象剪枝

**状态**: draft
**类型**: requirement
**优先级**: high

## 背景

Pluse 现在已经有多种会随工作推进而产生的对象：

- `Quest`
- `Run`
- `Todo`
- `SessionCategory`
- `QuestOp`
- `ProjectActivity`

这些对象虽然都和 `Quest` 有不同程度的关系，但它们并不属于同一种语义：

- 有些是 `Quest` 本体
- 有些是历史记录
- 有些是派生出来的人类注意力信号
- 有些只是引用了 `Quest`

当前系统最大的问题不是“缺少更多清理动作”，而是**缺少明确的生命周期归属规则**。

这会直接带来一个已经暴露出来的体验问题：

- Quest 被归档后，由它产生的 `review` Todo 仍然留在工作面里
- 这些 Todo 会继续占据人类注意力，即使源 Quest 已经退出当前工作面
- 同时，普通 Todo 又不应该因为 Quest 归档被一起清掉

这说明当前缺的不是“再加一条 if”，而是：

- 哪些对象会跟随 Quest 生命周期收敛
- 哪些对象只是引用 Quest，但仍然保持独立
- 这些规则应该由哪一层系统负责

## 顶层目标

Pluse 需要提供一套清晰的生命周期归属规则，使系统能够在 Quest 生命周期变化时，正确收敛由它派生的低价值噪音，同时保留真正独立的人类工作项和历史记录。

这项能力的目标不是做通用级联删除引擎，而是先把：

- Quest
- Quest 派生的人类注意力信号
- Quest 的历史与上下文记录

这三者之间的关系厘清。

## 核心问题

### 1. `originQuestId` 只是引用，不等于生命周期归属

当前 Todo 可以记录 `originQuestId`，但这只说明“这个 Todo 和某个 Quest 有来源关系”，并不能自动推出：

- Quest 归档时 Todo 应一起归档
- Quest 恢复时 Todo 应一起恢复
- Quest 删除时 Todo 应被强制清理

如果把所有 `originQuestId` 都视为级联关系，普通人工 Todo 也会被误伤。

### 2. 系统缺少“派生对象”和“独立对象”的稳定区分

当前至少已经存在两类不同的 Todo：

- 人工建立、表示真实承诺或待办的普通 Todo
- 由系统或 hooks 根据 Quest / Run 自动生成的 `review` Todo

这两类 Todo 虽然都长在 `Todo` 模型里，但生命周期语义并不一样。

若系统不能区分：

- 哪个是独立工作项
- 哪个只是派生出来的 review 信号

就无法做正确的剪枝。

### 3. 归档不会自动等于一切都要跟着消失

Quest 归档后，不同关联对象应有不同命运：

- Run 历史应保留
- QuestOp / Activity 历史应保留
- 普通 Todo 默认应保留
- 但系统生成的 review Todo 应退出当前工作面

如果没有这层区分，系统就会在“什么都不清”和“清得过头”之间摇摆。

### 4. 当前系统缺少明确的生命周期主控层

现在 review Todo 的生成路径既可能来自：

- Quest / Run 收尾逻辑
- hooks 策略

但 cleanup 的责任不应该因此落到 hooks 上。

系统需要明确：

- 谁拥有 Quest 生命周期
- 谁负责根据 Quest 生命周期去收敛派生对象
- hooks 负责“产生什么信号”，还是也负责“何时清理信号”

如果这层边界不明确，后续会不断把 cleanup 逻辑散落到各处。

## 核心需求

### 1. 系统必须建立 Quest 关联对象的生命周期归属规则

至少要能明确区分：

- Quest 本体
- Quest 历史
- Quest 派生的注意力信号
- 只是引用 Quest 的独立对象

### 2. Quest 归档时必须能收敛派生的 review 信号

当 Quest 被归档后，系统必须支持把由该 Quest 派生出来、只用于提醒人类 review 的 Todo 收敛掉。

### 3. 普通 Todo 不应因 Quest 归档被误清理

即使某条 Todo 记录了 `originQuestId`，只要它代表的是独立工作项，而不是派生 review 信号，就不应被 Quest 归档自动剪掉。

### 4. Quest 恢复不应默认恢复已被剪枝的 review Todo

被剪枝的 review Todo 本质上是“旧的注意力信号”。

Quest 恢复后，系统不应把这些旧信号自动带回工作面，否则会重新制造噪音。

### 5. 生命周期剪枝规则必须有明确主控层

Quest 归档引发的派生对象清理，应由拥有 Quest 生命周期的系统层负责，而不是散落在 hooks、前端或临时脚本里。

### 6. 当前设计应先解决最小闭环，而不是提前抽象出通用引擎

系统目前首先需要解决的是：

- Quest 归档时 review Todo 如何被正确收敛

而不是立刻引入：

- 通用对象依赖图
- 任意对象生命周期策略 DSL
- 通用清理编排平台

## 产品边界

- **Pluse 负责**
  - 定义哪些对象是生命周期主对象
  - 定义哪些对象属于派生信号
  - 在主对象生命周期变化时做正确收敛

- **hooks 负责**
  - 决定是否产生 review 信号
  - 决定在什么 run 事件上产生这些信号

- **Human / Agent 不负责**
  - 手工收拾系统本应自动剪掉的 review 噪音

## 不在范围内

当前 requirement 不规定：

- 是否引入新的数据库字段来表达“派生关系”
- 是否做通用 lifecycle policy engine
- Project / Domain 归档时是否联动 Quest / Todo
- Run 历史是否做额外清理
- UI 如何展示“被系统剪枝”的 Todo
- 未来其他派生对象的完整剪枝策略

这些属于后续 design / spec。

## 成功状态

如果这个需求被满足，系统应具备以下可观察特征：

- Quest 归档后，不再残留属于它的过时 review 信号
- 普通 Todo 不会因为 Quest 归档被误清理
- Quest 恢复后，不会把旧 review 噪音重新带回
- 系统内部对“谁拥有哪类对象的生命周期”有清晰边界
- 后续做 Todo / Session 剪枝时，不再需要每次从个案重新拍脑袋决定

## 失败状态

如果后续设计或实现仍然出现以下情况，说明这个需求没有被满足：

- Quest 归档后 review Todo 继续堆在工作面里
- 普通 Todo 因为 `originQuestId` 被一并误删或误归档
- Quest 恢复时旧 review Todo 被自动复活
- cleanup 规则继续散落在 hooks、runner 和临时 patch 中
- 每新增一种派生对象，都要重新猜一次“它到底该不该跟着 Quest 收敛”
