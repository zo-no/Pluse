# 0004 — Automation IA And Naming

**状态**: draft
**类型**: design
**关联 requirement**:
- `docs/v2/requirements/0004-ai-task-result-handling.md`
- `docs/v2/requirements/0001-human-workload-control.md`

## 设计目标

这份 design 解决的不是“把几个中文文案改掉”，而是：

- 用更稳定的产品语义承接 `Quest(kind='task')`
- 让用户在界面上清楚区分“系统执行工作”和“人类待办”
- 为后续的结果处理语义留出清晰位置

目标不是重做底层 Quest 模型，而是先把用户面和信息架构收敛到正确方向。

## 非目标

本 design 当前不讨论：

- 数据库字段改名
- HTTP / CLI 接口改名
- Quest / Run / Todo 的底层数据结构重构
- 结果处理语义的具体字段设计
- 视觉样式与组件级 UI 细节

这些内容留给后续 spec。

## 核心设计判断

### 1. 用户面不再使用 “AI 任务 / 人类任务”

产品语义统一收敛为：

- `会话`
- `自动化`
- `待办`

原因：

- `Quest(kind='task')` 已经不只承载 AI prompt，也承载 script executor
- `Todo` 的本质是人类待办，不是“另一种 task”
- “AI 任务 / 人类任务”会把两种不同对象硬塞进同一语义家族

### 2. “自动化”描述的是执行方式，不是触发方式

一个对象是否叫“自动化”，不取决于它是不是自动开始，而取决于：

- 触发之后是否主要由系统执行
- 是否是一个可被保存、复用、调度的执行配置

因此手动点击触发的 `Quest(kind='task')` 仍然属于自动化。

### 3. `Quest(kind='task')` 继续作为内部技术边界

当前不把内部 `task` 全量改名为 `automation`。

原因：

- 现有底层模型已经稳定
- 这次要解决的是产品语义和 IA，不是数据库主对象命名
- 过早改动内部命名会带来高迁移成本，但不直接提升用户体验

用户面与内部技术名词可以继续保持分离：

- 内部：`Quest(kind='task')`
- 用户面：`自动化`

## 能力结构

### 1. 顶层对象分工

用户面应明确看到三类对象：

- `会话`
  - 持续对话与上下文推进

- `自动化`
  - 由系统执行的工作单元
  - 可以手动触发，也可以被调度触发

- `待办`
  - 需要人类处理的独立事项
  - 可以来自自动化，也可以独立存在

### 2. 自动化的主分类维度

自动化的一级分类不按 executor 类型，而按触发/调度方式表达：

- `手动`
- `定时`
- `周期`

对应当前内部模型：

- `scheduleKind='once'` → 用户面显示为 `手动`
- `scheduleKind='scheduled'` → 用户面显示为 `定时`
- `scheduleKind='recurring'` → 用户面显示为 `周期`

这里明确做一个设计决议：

- 内部 `once` 继续保留
- 用户面不显示“一次性”
- 用户面正式显示为“手动”

这样可以避免它与“定时一次”在用户心智中混淆。

### 3. executor 类型降为二级信息

自动化的 `executorKind`（例如 `ai_prompt` / `script`）不是顶层 IA 的一级分类。

它应作为：

- 详情页配置项
- 卡片次级信息
- 筛选器中的附加条件

而不是与 `待办 / 自动化` 并列的主导航维度。

原因：

- 用户首先关心“这是谁来做”和“它怎么触发”
- 不首先关心“它内部是 prompt 还是 script”

## 信息架构

### 1. 右侧统一工作面板

当前“任务面板”在用户面应重构为“待办 + 自动化”的统一工作面。

顶层筛选收敛为：

- `全部`
- `待办`
- `自动化`

不再使用：

- `全部 / 人类 / AI`

### 2. 新建入口

“新建任务”不再作为模糊总入口暴露。

应改为明确的两类动作：

- `新建待办`
- `新建自动化`

如果仍保留统一弹窗，也应先让用户明确选择对象类型，再进入对应表单。

### 3. 详情页口径

`Quest(kind='task')` 的详情页在用户面统一显示为：

- `自动化`

`Todo` 的详情页统一显示为：

- `待办`

与自动化相关的动作文案同步收敛：

- `转为自动化`
- `新建待办`
- `立即运行`

### 4. 自动化卡片的信息层级

自动化卡片至少要同时表达三类信息，但不能混成一个状态词：

1. **类型信息**
   - 手动 / 定时 / 周期

2. **执行信息**
   - 运行中 / 失败 / 已取消 / 空闲

3. **结果信息**
   - 是否有新的待处理结果
   - 是否已经处理完

其中：

- 类型信息回答“它是什么自动化”
- 执行信息回答“它现在跑得怎样”
- 结果信息回答“用户还要不要关注这次产出”

`0004-ai-task-result-handling` 的后续设计，应主要落在第 3 层，而不是继续挤压前两层。

## 状态语义的呈现原则

### 1. 不再用单一状态词承载全部语义

自动化在用户面至少同时存在三条正交语义：

- 触发/调度方式
- 当前执行状态
- 结果处理状态

后续设计不应再试图用一个 badge 或一个列表分区，强行承接三件事。

### 2. `run completed` 不驱动自动化退出关注面

自动化成功执行一次，不应自动让用户感知它已经“被处理掉”。

界面上的退出或降级，应优先受以下因素影响：

- 结果是否仍待关注
- 是否存在失败/异常
- 是否只是等待下一次运行

这条原则与 `0004-ai-task-result-handling` 保持一致。

## 兼容与清理边界

### 1. 本期保留的内部概念

以下概念继续保留，不视为冗余：

- `Quest.kind='task'`
- `scheduleKind`
- `Run.trigger`
- `executorKind`

它们分别表达不同层次的问题：

- 对象身份
- 调度配置
- 单次运行的触发来源
- 执行器类型

### 2. 本期可以进入清理名单的内容

以下更接近历史残留或表层债务：

- “AI 任务 / 人类任务”文案
- “全部 / 人类 / AI”这类筛选命名
- 未导出的旧 `task` 类型定义
- 仍沿用旧任务口径的历史文档和组件注释

### 3. 本期不做内部文件级大改名

例如 `TaskDetail` / `TaskComposerModal` 这类内部文件名，当前不要求同步改名。

原因：

- 设计目标是先修正产品语义
- 文件级重命名属于工程清理，不应阻塞 IA 收敛

## 分期建议

### Phase 1: 用户面命名收敛

先统一：

- 自动化 / 待办 文案
- 顶层筛选文案
- 新建入口文案
- 详情页 kicker 和动作文案

目标是先让用户心智与对象边界对齐。

### Phase 2: 自动化信息层级收敛

在命名稳定后，再收敛自动化卡片与详情页里的信息层级：

- 类型信息
- 执行信息
- 结果信息

目标是避免“一个状态词承载全部语义”。

### Phase 3: 工程清理

最后再处理：

- 死类型
- 过时文档
- 不再使用的旧命名

这期不改变产品能力，只消除歧义和技术债。

## 关键取舍

### 1. 先改用户面，不先改数据库名

如果一开始就推进底层大改名，工程成本会远高于产品收益。

当前最重要的是：

- 先让用户看到正确的对象边界
- 再决定是否需要进一步收敛内部命名

### 2. 先按触发方式分类，不按 executor 分类

对用户来说：

- 手动 / 定时 / 周期

比：

- AI prompt / script

更接近任务管理与注意力分配语义。

### 3. 把结果处理语义留给后续专门设计

这份 design 不试图直接解决“已处理”怎么落字段。

它只先保证：

- 自动化不会继续被错误叫成 AI 任务
- 待办不会继续被错误叫成人类任务
- 后续结果处理设计有稳定的 IA 落点
