# 0006 — 会话列表一键复制 ID 设计方案

**状态**: draft
**类型**: design
**关联 requirement**: `docs/v2/requirements/0006-quick-select-and-copy-session-id.md`

## 设计目标

将 `Quest` 会话列表中的 `quest.id` 获取动作从“打开会话 → 选中复制”降到“列表内单步完成”。

## 设计立场

- `Low-friction`
  - 目标是让“拿 ID”尽可能接近零认知成本
- `List-local`
  - 在列表行内完成，不要求打开详情页
- `No scope creep`
  - 本期只做单条会话 ID 复制，不做批量、快捷键、或消息发送链路扩展
- `UI + Feedback`
  - 可见反馈是动作可用性的必要条件

## 能力边界

- 作用对象：`Quest(kind='session')` 列表中的会话项
- 支持动作：复制该会话的 `id`
- 支持范围：当前项目视图下展示的列表行
- 不支持范围：任务态会话、批量选择、列表排序与筛选重构、消息发送重构

## 交互模型（最小闭环）

### 1. 列表行为

- 每条会话在右上角/尾部提供一个 `Copy ID` 操作（图标按钮或文字按钮）。
- 点击动作时复制对应会话 `id` 到剪贴板。
- 触发后显示成功反馈（例如行内文案变更或 toast）。
- 反馈需在 1~2 秒后恢复默认状态。

### 2. 失败兜底

- 系统无剪贴板权限或调用失败时，给出失败提示。
- 失败后仍保留可见 ID 文本，供用户手动复制。

### 3. 可用性细节

- 按钮应有明确可点击目标（非仅文本命中）。
- 支持键盘可聚焦焦点访问。
- 不遮挡会话主要信息。

## 能力与对象关系

本设计假设前端在已有会话列表模型中已经拿到了 `Quest` 对象的 `id`：

- 行为只读，不改变会话本体状态。
- 不引入新对象、不会改变 `Quest` 生命周期。
- 不依赖 `session-category`、`todo`、`run` 等新状态。

## 方案取舍

### 方案 A（本期推荐）：每行内按钮

- 优点：实现成本低、学习成本低、行为明确。
- 缺点：按钮占用行内空间。
- 适配本需求优先级和约束（“一个按钮”）最适配。

### 方案 B：行内 hover 后才显示

- 优点：界面更干净。
- 缺点：可发现性依赖交互手势，误触更低但新手找不到入口。

### 方案 C：列表右侧统一顶部复制区

- 优点：统一视觉入口。
- 缺点：需要额外选择状态，背离“一个按钮、直达会话”目标。

### 选型

采用方案 A（单条行内按钮），满足“单按钮、最少步骤”的当前需求。

## 依赖与实现前置

- UI 层需支持剪贴板写入 API 与降级文案。
- `quest.id` 已在列表项中完整展示（或可从列表行数据直接读取）。
- 现有列表渲染应允许轻微新增交互控件。

## 分期说明

本 design 为单期交付；不依赖后端变更、数据库迁移、后续 API 变更。

## 不在范围内

- CLI 命令层新增参数
- 批量会话操作
- ID 格式改造
- 会话发送消息链路优化

## 验收要点

- 单条会话可在列表视图中用一个按钮复制 ID。
- 复制成功有明确提示。
- 未复制成功时有清晰错误提示且可回退到手动复制。
