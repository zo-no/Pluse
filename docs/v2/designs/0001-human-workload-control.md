# 0001 — Human Workload Control Design

**状态**: draft
**类型**: design
**关联 requirement**: `docs/v2/requirements/0001-human-workload-control.md`

## 设计目标

这个 design 要回答的不是“Todo 怎么改几个字段”，而是：

- Pluse 如何在不拖慢 Agent 的前提下，控制人类负载
- Pluse 如何把人类参与压缩成高价值、低摩擦、可承受的交互
- Pluse 如何逐步把重复、低意义劳动迁移给 AI

## 设计立场

这个设计建立在以下立场上：

- `AI-first`
  - Agent 是默认主推进者，而不是等待人类逐条驱动的执行器

- `Human-exception`
  - 人类参与是高价值例外，不是默认执行面

- `Dual-speed collaboration`
  - Agent 与人类是两种不同速度、不同职责、不同带宽的用户
  - 系统必须承担两者之间的转换与缓冲

- `Progressive evolution`
  - 不追求一次性把所有工作自动化
  - 先做基础承接层，再做乘数层，最后做动态演化层

## 非目标

这份 design 当前不讨论：

- 具体数据库字段
- HTTP / CLI 接口名称
- UI 视觉稿
- provider 级执行细节

这些内容留给后续 spec。

## 核心设计思路

Pluse 不应把“任务管理”和“AI 运行”看成两条平行线，而应把系统设计成：

- 一个面向 Agent 的`高速执行面`
- 一个面向人类的`高信号决策面`
- 一个位于中间的`工作负载控制与交接层`

真正的设计核心不在“列出更多信息”，而在：

- 先由系统吸收复杂度
- 再决定什么应该继续由 Agent 处理
- 什么应该被托管或周期化
- 什么值得上浮给人
- 上浮给人时应该如何包装

## 能力分层

### 基础层

基础层解决“系统能不能先接住复杂度”。

#### 1. 上下文连续层

系统需要稳定承接：

- Quest 上下文
- 用户偏好与长期方向
- 历史判断与近期状态
- Agent 执行过程中已经获取的工作背景

这层的目标不是做长期记忆功能，而是避免：

- 重复解释
- 重复授权
- 重复给出同类判断

#### 2. 高质量交接层

系统需要把 Agent 内部复杂度先消化，再向人类输出最小交互单元。

人类看到的应是：

- 缺什么信息
- 需要什么审批
- 需要做哪个关键判断
- 为什么现在值得找人

而不应是：

- 原始日志
- 零散 Todo 洪水
- 未经整理的 review 列表

#### 3. 双速协作层

系统必须显式区分两类交互面：

- `Agent side`
  - 连续推进
  - 少等待
  - 少人工确认

- `Human side`
  - 低打断
  - 高信号
  - 高价值参与

这意味着后续很多对象不应被同一套默认交互承载。

### 乘数层

乘数层建立在基础层之上，用于放大整体收益。

#### 4. 负载控制层

系统需要控制的不是任务数量，而是人类负载。

这层至少要承担：

- 决定什么进入当前工作面
- 决定什么延后、降级、等待
- 保护长期任务不被低价值紧急事项持续挤掉
- 限制低价值打断的默认打断权

#### 5. 重复劳动迁移层

系统需要逐步识别：

- 周期任务
- 模板化流程
- 一键触发动作
- 可默认托管的低风险流程

这层的目标是让“人手动重复执行”逐步转化成：

- 周期化
- 参数化
- 托管化

#### 6. 人类交互包装层

这层和高质量交接相关，但更强调“交给人之后该长什么样”。

设计目标是让人类参与变成：

- 补一个信息
- 选一个方向
- 审一个决定
- 处理一个异常

而不是让人接管整段流程。

### 后置演化层

后置演化层不是第一阶段阻塞项，但它决定系统是否会长期变强。

#### 7. 动态教练层

系统在必要时进入教练模式，但训练内容必须是动态的。

系统训练人的依据应来自当前瓶颈，例如：

- 人类最常卡住 Agent 的步骤
- 最常重复出现的低质量输入
- 最影响推进效率的判断短板

#### 8. 托付与纠偏层

系统要让用户敢于逐步减少控制，就必须同时提供：

- 可托付性
- 可解释性
- 可纠偏性

否则再强的自动化也不会真正被放权。

## 关键角色

### Agent

Agent 的默认职责是：

- 持续推进 Quest
- 吸收上下文
- 尽量自行完成低风险工作
- 只在确有必要时请求人类参与

### Human

Human 的默认职责是：

- 提供方向
- 提供关键判断
- 提供必要信息
- 承担关系与承诺层的决策

Human 不应再承担大量机械收尾工作。

### Control Plane

Pluse 本身需要承担一个控制平面的角色，用来：

- 承接 Agent 侧复杂度
- 过滤和包装人类交互
- 管理当前工作面
- 决定托管、等待、延后和上浮

## 分期建议

### Phase 1: 基础承接层

优先建立：

- 上下文连续层
- 高质量交接层
- 双速协作层的基本分界

目标是先解决：

- 重复解释
- 手动收拾残局
- Agent 把复杂度直接甩给人

### Phase 2: 负载控制与重复劳动迁移

在基础承接层成立后，再推进：

- 负载控制层
- 重复劳动迁移层
- 人类交互包装层

目标是解决：

- 低价值打断
- 长期任务被持续挤掉
- 重复劳动长期留在人手里

### Phase 3: 动态演化

最后再推进：

- 动态教练层
- 托付与纠偏层的增强

目标是让系统从“能减负”逐步走向“会共演化”。

## 关键取舍

### 1. 先做承接层，不先做聪明排序

如果系统还不能承接上下文、不能包装交接，那么更聪明的排序只会让混乱看起来更有组织。

### 2. 先解决 Agent 被人卡住，再谈高级教练能力

动态训练很有价值，但它建立在前面的基础层和乘数层已经成立的前提上。

### 3. 先做高杠杆平台能力，不追单点功能胜利

高质量交接层和上下文连续层都属于平台能力，它们一旦成立，会同时加速多个需求。

## 设计完成标准

这份 design 达到可进入 spec 的标准应是：

- 已明确能力分层
- 已明确角色分工
- 已明确 phase 顺序
- 已明确哪些问题属于基础层，哪些属于乘数层，哪些属于后置演化层
- 尚未落入字段、接口、UI 等实现细节
