## 设计目标

这份 design 解决一件事：

- 如何让 Pluse 的自动化按项目逐步形成闭环，同时不把更多信息压力推给用户

它不是自动化控制台的实现 spec，也不是所有项目的 prompt 模板集合。

## 设计立场

### 1. 项目是自动化闭环的默认边界

项目自动化默认只处理本项目：

- 读取本项目 Quest / Run / Todo / Reminder / 记录
- 判断本项目是否偏航
- 生成本项目提醒
- 写入本项目记录

这保证「生活」「财务」「内容」「研发」不会互相污染。

### 2. 跨项目总控只管理注意力

跨项目总控不替项目执行具体工作。

它只负责：

- 当前主线项目判断
- 提醒优先级调整
- 非主线项目降噪
- 跨项目摘要

当前最合适承接总控的项目是「自我对话」，因为它已经是默认入口，也已经承担真实需求澄清和主线判断职责。

### 3. 自动化默认静默，提醒是升级后的结果

自动化运行本身不是提醒。

运行结果的默认流向是：

```text
Run output
  -> 静默记录
  -> 摘要
  -> 提醒
  -> 强干预
```

只有越过透出门槛的内容，才进入 Reminder。

### 4. 右侧栏只承接人类行动，不承接自动化全量状态

右侧栏应该继续保持高信号：

- 当前需要用户处理的提醒
- 少量异常或阻塞
- 当前项目的下一步动作

自动化管理应进入独立控制台或项目内自动化面板。

## 核心模型

### 1. Project Loop

每个项目的自动化闭环由四步组成：

```text
Observe -> Decide -> Act -> Record
```

- `Observe`: 读取项目状态、近期运行、提醒、待办、记录
- `Decide`: 判断是否偏航、是否缺事实、是否需要用户
- `Act`: 创建提醒、触发低风险动作、提出决策点
- `Record`: 写入运行摘要，供后续自动化复用

### 2. Control Loop

跨项目总控只做注意力调度：

```text
Collect project signals -> Pick mainline -> Allocate attention budget -> Surface exceptions
```

它不直接创建其他项目的执行计划，除非输出的是跨项目层面的提醒，例如「今天主线是财务管理」。

### 3. Surface Gate

所有自动化输出都要经过透出门槛：

| 等级 | 名称 | 行为 |
| --- | --- | --- |
| `silent` | 静默 | 只保留在 Run / 项目记录中 |
| `digest` | 摘要 | 进入日报或周报，不即时打扰 |
| `attention` | 提醒 | 创建 Reminder，等待用户处理 |
| `blocking` | 强干预 | 用高优先级提醒暴露异常或关键决策 |

默认等级应是 `silent`。

## 自动化成熟度

| 等级 | 名称 | 允许行为 | 升级条件 |
| --- | --- | --- | --- |
| `L0` | 配置中 | 只记录想法和候选流程 | 项目目标和闭环周期明确 |
| `L1` | 静默运行 | 可周期运行，但不创建提醒 | 连续运行稳定，输出不重复 |
| `L2` | 摘要运行 | 可进入日报/周报摘要 | 摘要对项目复盘有用 |
| `L3` | 提醒运行 | 可创建少量项目提醒 | 确认提醒确实提高推进质量 |
| `L4` | 强干预 | 可在严重偏航时高优先级触达 | 仅主线项目或高风险项目允许 |

任何项目默认从 `L0` 或 `L1` 开始，不直接进入 `L3/L4`。

## 当前项目接入判断

| 项目 | 当前建议等级 | 建议闭环 | 透出策略 |
| --- | --- | --- | --- |
| 生活 | `L3` 试点 | 晨间计划、午间偏航、晚间复盘、睡前收束、每日记录 | 每天最多 2 条主动提醒，成功运行静默 |
| 自我对话 | `L1` 候选 | 主线判断、跨项目摘要、注意力预算 | 先静默，不直接管理项目内部行动 |
| 财务管理 | `L0 -> L1` | 周度盘点、月度资产/支出复盘、决策前复核 | 低频摘要，重大决策才提醒 |
| AI硬核自媒体 | `L0 -> L1` | 选题、草稿、发布、反馈复盘 | 主线时提醒，非主线进摘要 |
| 视觉艺术自媒体 | `L0` | 选题、拍摄、剪辑、发布节奏 | 暂不主动提醒 |
| 2-3-曲率同调_小说计划 | `L0` | 写作节奏、设定一致性、章节推进 | 低频摘要 |
| Pluse | `L0 -> L1` | 需求、设计、spec、实现状态检查 | 不自动编码；阻塞或失败才提醒 |
| Closer | `L0` | 产品假设、原型推进、用户理解 | 先不提醒 |
| pr-review-agent | `L1` | PR 队列、失败检查、审阅产物 | 失败或需要人工合并才提醒 |
| 网易云CLI | `L0` | 工具维护、低风险脚本检查 | 默认静默 |

## 分期计划

### Phase 0: 固定规则与生活试点

目标：

- 只维护生活项目
- 不再继续扩其他项目自动化
- 观察 7 天提醒质量和重复率

规则：

- 生活每天最多 2 条主动提醒
- 自动化成功运行不提醒
- 重复语义当天合并
- 不创建 Todo
- 不跨项目管理

### Phase 1: 自我对话总控静默运行

目标：

- 每天判断主线项目
- 生成跨项目摘要
- 不直接创建项目内部行动

输出先保持 `silent` 或 `digest`，确认有效后再允许创建少量主线提醒。

### Phase 2: 财务管理低频接入

目标：

- 每周盘点财务现实
- 每月做资产/支出复盘
- 大额消费或投资前提醒复核

财务自动化不做自动交易、不做最终投资判断，只做事实整理和决策提醒。

### Phase 3: 内容项目接入

目标：

- 先接入 AI硬核自媒体
- 再根据主线状态接入视觉艺术自媒体

内容项目的提醒只在项目成为主线时提高权重。

### Phase 4: 研发项目接入

目标：

- Pluse 自动化只做需求、设计、spec、失败和阻塞检查
- Closer 先做产品假设和原型推进检查
- pr-review-agent 保持低噪音失败提醒

研发自动化不绕过 Pluse v2 的 requirement / design / spec 流程。

### Phase 5: 自动化控制台

目标：

- 从右侧栏中移出自动化管理压力
- 建立按项目聚合的自动化控制台
- 支持查看运行状态、下次运行、失败、透出策略和启停操作

## UI 方向

### 1. 顶部入口

用户面保留清晰对象入口：

- 待办
- 提醒
- 自动化

自动化需要成为独立管理面，而不是隐藏在任务栏里。

### 2. 右侧栏

右侧栏只保留：

- 人类需要处理的提醒
- 当前项目关键行动
- 少量自动化异常

不展示所有自动化运行输出。

### 3. 自动化控制台

自动化控制台按项目聚合：

| 项目 | 自动化健康 | 下次运行 | 最近结果 | 透出策略 | 操作 |
| --- | --- | --- | --- | --- | --- |
| 生活 | 正常 | 21:30 | 今日轻微偏航 | attention | 查看 / 暂停 |
| 财务管理 | 待配置 | 周日 20:00 | - | digest | 配置 |
| Pluse | 有异常 | 每日 10:00 | 上次失败 | blocking | 查看 |

点开项目后，再进入项目内自动化列表。

## 进入 spec 前需要确认

在写实现 spec 前，需要先确认：

1. `surfacePolicy` 是先放在 prompt 规则里，还是进入结构化字段
2. `L0-L4` 是否需要成为系统字段，还是先作为文档和 UI 口径
3. 自动化控制台第一期是全局页，还是项目页内分区
4. 每日提醒预算先由 Agent prompt 约束，还是进入 Reminder ranking 逻辑
5. 自我对话总控是否就是长期跨项目控制平面
