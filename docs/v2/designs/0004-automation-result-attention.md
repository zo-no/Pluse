# 0004 — Todo / Automation Workspace IA

**状态**: draft
**类型**: design
**关联 requirement**: `docs/v2/requirements/0004-ai-task-result-handling.md`
**关联 design**: `docs/v2/designs/0004-automation-ia-and-naming.md`

## 设计目标

这份 design 只解决一件事：

- 如何让右侧工作面板真正体现 `待办` 与 `自动化` 的边界

重点不是发明新的底层状态机，而是纠正当前工作面板的产品语义和信息架构。

## 非目标

本 design 不讨论：

- result attention 新模型
- notification / inbox
- schema 迁移
- API / CLI 改名
- `Quest.status` 的底层枚举重构

## 当前问题

当前右侧工作面板有三处错位：

### 1. scope 视角错位

面板顶部仍然提供：

- `全局`
- `项目`
- `会话`

其中 `会话` 不是一个合适的主工作视角，因为用户不会一次只处理一个会话里的工作项。

### 2. 生命周期语义错位

当前面板会把自动化和待办一起塞进：

- active list
- history list

然后使用：

- `已完成`
- `已处理`

这类标签去描述混合对象。

这会把自动化错误地拖进人类待办语义。

### 3. 自动化缺少自己的组织方式

自动化本身已经具备一套独立维度：

- `scheduleKind`
- 运行状态
- 是否异常

但工作面板没有用这些维度组织自动化，只是把它们当成“另一种任务卡片”。

## 核心设计判断

### 1. 工作面板是“工作项面板”，不是“会话面板的补充”

因此它的主视角应围绕：

- 全局
- 项目
- 待办
- 自动化

而不是围绕当前会话。

### 2. Todo 与 Automation 共处一个面板，但不共享历史语义

Todo 继续保留：

- `待处理`
- `已完成`
- `已取消`

Automation 改为保留：

- `待触发`
- `运行中`
- `已执行`
- `失败`
- `已取消`

这里的 `已执行` 只表示最近一次单次运行结束，不表示对象“已完成”。

### 3. 自动化不进入“已处理”区

`已处理` 是人类工作流语义。

因此工作面板中：

- Todo 可以有历史区
- Automation 不应因为 `done` 落入历史区

自动化应该始终以“仍存在的自动化对象”被组织，只在归档时离开主视图。

### 4. 自动化按自动化语义分组

自动化主分组采用以下优先级：

1. `运行中`
2. `异常`
3. `周期`
4. `定时`
5. `手动`

分组规则：

- `activeRunId` 或 `status='running'` -> `运行中`
- `status='failed' | 'cancelled'` -> `异常`
- `scheduleKind='recurring'` -> `周期`
- `scheduleKind='scheduled'` -> `定时`
- 其余 -> `手动`

这样可以同时满足：

- 看见自动化当前是否活跃
- 看见自动化属于哪种触发方式
- 不再依赖“已完成/已处理”来驱动位置变化

## 视图结构

## 1. 顶部 scope

保留：

- `全局`
- `项目`

移除：

- `会话`

## 2. 对象筛选

保留：

- `待办`
- `自动化`

这是对象类型视角，不是生命周期视角。

## 3. 内容组织

### 当包含 Todo 时

`项目` 视图下，Todo 分为：

- `待办`
- `历史`
- `归档`

`全局` 视图下，Todo 先按 `项目` 分组，再在项目组里展示待办与历史。

当前不在这一期直接引入 `领域 -> 项目` 双层分组，后续再看是否需要上浮到领域层。

### 当包含 Automation 时

Automation 分为：

- `运行中`
- `异常`
- `周期`
- `定时`
- `手动`
- `归档`

用户在 `待办 / 自动化` 两种对象视图之间切换，不再需要一个“全部混合流”。

## 4. Domain Sidebar

左侧 `领域` 视图下，领域分组默认展开，优先让用户直接看到项目，而不是先要求用户逐个手动展开。

## 状态文案

## 1. Automation

映射为：

- `pending` -> `待触发`
- `running` -> `运行中`
- `done` -> `已执行`
- `failed` -> `失败`
- `cancelled` -> `已取消`

## 2. Todo

继续保持：

- `pending` -> `待处理`
- `done` -> `已完成`
- `cancelled` -> `已取消`

## 对现有模型的影响

这次设计不要求调整：

- `Quest.kind='task'`
- `Quest.status`
- `scheduleKind`
- `reviewOnComplete`

只改变用户面如何解释这些字段。

## 分期判断

本 design 对应的首轮实现应只覆盖：

- 右侧工作面板分组
- 自动化状态文案收敛
- 会话 scope 移除

不在本期继续下探：

- 自动化结果关注层
- 额外 attention badge
- 新的统计口径
